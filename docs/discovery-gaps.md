# 前期需求调研缺口分析 (Discovery Gaps)

> 背景：现有产物盘点
> - ✅ 方案设计侧扎实：PRD、6 份 ADR、#4 技术设计、PM/Designer/Engineer Top5 创意
> - ⚠️ 问题验证侧薄弱：痛点、可行性、采纳意愿多为经验假设，缺实证
>
> 本文从 PM / Designer / Engineer 三视角各发 5 项"前期调研还缺的准备"，综合排序 Top 5，逐条给出待验证假设与最小验证方式。
> 适用阶段：Initial Discovery 的**验证收尾**--在动手写代码前，用最小成本证伪/证实关键假设。

---

## 0. 核心判断

现有文档回答了"**怎么做**"（架构、ADR、技术设计），但"**该不该做、能不能成、会不会被用**"这三类前置问题尚未用证据回答。具体表现为：
- 痛点只有定性描述，无量化基线（SRE 每天被求助几次？深层排查占比多少？MTTR 到底多少？）。
- "LLM 能把大白话转成准确 QL"这一**产品命门假设**完全未做可行性验证。
- 无真实数据样本（实际 metric/label、真实故障工单、历史 RCA），导致技术设计里的元数据缓存、剧本库、知识库都是"空中楼阁"假设。
- 无竞品对标、无安全合规审查、无目标用户画像。

下面 15 项缺口围绕"补齐证据链"展开。

---

## 1. PM 视角：商业价值 / 战略对齐 / 客户影响

### PM-G1 量化痛点基线（Pain Point Baseline）
现状：PRD 称"多对一人力灾难""MTTR 居高不下"，但**全无数字**。需采集：SRE 每月被求助工单数、平均处理时长、深层排查占比、因 QL 门槛放弃自助的比例、当前 MTTR。没有基线，brainstorm PM-5 的 ROI 仪表盘无参照系，也无法在事后证明"降本增效"。

### PM-G2 竞品与替代方案对标（Competitive Landscape）
现状：零竞品分析。需对标 AIOps 类产品（Datadog Watchdog、PagerDuty AIOps、New Relic AI）、开源方案（k8sgpt、robusta、holmesgpt）、以及"用通用 ChatGPT 贴日志"的野生做法。明确我们的差异化点（私有 LGTM 栈 + 原生 Tool-Calling + RCA 沉淀），避免重复造轮子或定位撞车。

### PM-G3 干系人共识与变更管理（Stakeholder Alignment）
现状：未确认 SRE 团队是否愿意"交出"排查权、开发是否愿意尝试、谁是执行赞助人。AI SRE 工具本质是改变工作流，阻力常来自"被替代焦虑"。需做干系人地图 + 至少 3 位 SRE、5 位开发的意愿访谈，否则上线即遇冷。

### PM-G4 MVP 成功标准定义（Success Criteria）
现状：brainstorm PM-5 提到 ROI 仪表盘，但未定义 MVP 的**验收阈值**。需明确：灰度期 deflection rate 目标（如 ≥30% 自闭环）、MTTR 缩短目标（如 ≥20%）、采纳率（如周活开发 ≥X）。无阈值则"做完了"无法判断"成了没"。

### PM-G5 商业模式与预算边界（Business Model & Budget）
现状：未明确这是内部工具还是可对外产品，也未框定 LLM 调用成本预算。需回答：内部降本工具 ROI 如何计算？单次排查 token 成本上限？是否走私有化 LLM（合规要求）？这直接决定技术选型（云端 API vs 本地模型）与 ADR-006 的可持续性。

---

## 2. Designer 视角：用户 / 体验 / 可用性

### D-G1 目标用户画像与排障旅程（Persona & Journey Map）
现状：PRD 只说"开发人员"，但开发分资深/初级、分业务域，排查能力差异巨大。需画像：谁在求助 SRE？求助前自己试了什么？在哪个环节放弃？触发求助的具体时刻？这决定 chat 交互的 prompt 引导、证据展示粒度、上手门槛设计。

### D-G2 真实排障工作流影子调研（Workflow Shadowing）
现状：所有交互设计基于想象。需跟班观察 3~5 次真实 SRE 排障全过程（从接到求助到给出结论）：他们先看什么、跨系统怎么切换、哪些信息是关键、卡点在哪。这直接喂给 brainstorm #3 剧本库（真实黄金路径而非臆想）和 D-1 Timeline 的信息层级。

### D-G3 现有监控面板与认知现状审计（Existing Dashboard Audit）
现状：PRD 说"排查依赖 SRE 预设面板"，但没盘点现有面板长什么样、开发能用懂多少。需收集现有 Grafana 面板清单、开发常用/看不懂的面板，找出"现有工具已能解决但没人会用"的 gap，避免产品做了 SRE 面板已有的事。

### D-G4 信任与采纳的可用性测试设计（Trust Usability Test Plan）
现状：brainstorm 把"信任"列为命门，但无验证设计。需设计：给开发看一份 AI 生成的 RCA（含/不含证据下钻），测其是否敢据此行动、是否会回查 SRE。用低保真原型即可测，验证 D-1/D-5 假设，无需等代码。

### D-G5 故障申告的自然语言语料采集（Utterance Corpus）
现状：PRD 举例 `"payment-service 刚才为什么大量 500？"`，但真实用户会怎么说？需采集 50~100 条真实/模拟的故障描述自然语言，分析句式、信息完备度、歧义程度。这直接验证"自然语言->QL"的意图识别可行性，也用于设计 prompt 与上手引导。

---

## 3. Engineer 视角：技术可行性 / 数据 / 集成

### E-G1 LLM QL 生成能力可行性验证（Feasibility Spike）⭐ 命门
现状：整个产品建立在"LLM 能把大白话转成准确 PromQL/LogQL/TraceQL"这一未验证假设上。需做最小 spike：取 20~30 个真实故障描述 + 真实 metric/label，让目标 LLM 生成 QL，人工评估准确率。若准确率低于阈值，整个产品地基动摇，应先解决（few-shot、元数据注入、剧本约束）再谈架构。

### E-G2 真实观测后端环境审计（Backend Environment Audit）
现状：技术设计假设了 Mimir/Loki/Tempo API 可达、metric 目录可拉取，但未核实。需审计：实际 API 版本与端点、鉴权方式（Token/mTLS/OIDC）、限流策略、网络可达性（Agent 部署位置能否访问）、现有 Grafana 数据源配置。这直接决定 tools 客户端实现与 ADR-004 的 timeout/重试策略。

### E-G3 真实 metric/label/log 样本采集（Real Data Samples）
现状：技术设计的 MetricCatalog（L2 校验）、system prompt 元数据注入都是假设，没有真实数据。需导出：实际 metric 名清单、高频 label、真实报错日志样本格式、真实 Trace 结构。没有这些，QL 校验器、缓存 key、prompt 都无法落地验证，单测也是空中楼阁。

### E-G4 历史故障工单与 RCA 语料库（Historical Incident Corpus）⭐ 飞轮前提
现状：brainstorm #1（知识库）和 #3（剧本库）都依赖历史 RCA 数据，但语料尚未收集。需拉取近 3~6 个月真实 incident 工单 + 对应 RCA，做：①故障模式分类（验证 H1：高频模式占比 ≥60%）；②RCA 结构化可行性评估；③"症状->根因"可复用性评估。这是验证 Top1/Top3 两个核心创意的前提，也是 PM-G1 基线的来源。

### E-G5 安全合规与数据敏感性审查（Security & Compliance Review）
现状：私有数据中心产品，但未审查 LLM 调用链路的数据合规。需明确：service 名/指标/日志/Trace 是否含敏感信息？发往云端 LLM 是否合规？是否必须本地化模型（vLLM/Ollama）？查询结果是否需脱敏？这直接影响 ADR-001（SDK 选型）、ADR-006（部署形态），是上线硬门槛。

## 4. 综合 Top 5 排序

**排序权重**：① 是否卡住产品命门（不验证就不能往下走）-> ② 影响面（牵动多个设计决策）-> ③ 验证成本（越低越优先做）。

| 排名 | 缺口 | 来源 | 一句话理由 |
|---|---|---|---|
| 🥇 #1 | LLM QL 生成能力可行性验证 | E-G1 | 产品命门假设未验证--若 LLM 转不准，一切架构白搭，必须最先 spike。 |
| 🥈 #2 | 历史故障工单与 RCA 语料库 | E-G4 | Top1 知识库 + Top3 剧本库 + PM-G1 基线的前提，一份数据验证多个核心假设。 |
| 🥉 #3 | 量化痛点基线 | PM-G1 | 无基线无法证明价值、无法定义成功、无法说服赞助人，是"该不该做"的最终裁判。 |
| #4 | 真实观测后端环境审计 + 真实数据样本 | E-G2+G3 | tools 实现、QL 校验、prompt 注入都依赖真实 API/数据，不审计则技术设计无法落地。 |
| #5 | 安全合规与数据敏感性审查 | E-G5 | 私有数据中心 + LLM 的硬合规门槛，决定能否用云端 API、是否必须本地模型，影响 ADR-001/006。 |

> 强力候选（未进 Top5 但建议并行）：D-G1 用户画像、D-G2 工作流影子调研（喂给剧本库与 Timeline 设计）、PM-G3 干系人共识（决定能否上线推广）。

## 5. Top 5 的理由与待验证假设

### #1 LLM QL 生成能力可行性验证（命门 Spike）
**理由**：产品的核心承诺是"大白话 -> 准确 QL"。若 LLM 在真实数据上生成准确率不足，再精巧的 ToolSpec/SafeToolExecutor 也无米之炊。这是**唯一一个"假失败则整个产品方向需调整"的假设**，必须用最小成本先证伪，而非等架构搭完才发现。

**待验证假设**：
- H1：目标 LLM 在注入真实 metric/label 目录后，单次 QL 生成准确率 ≥ 60%（再经自修正达 ≥ 80%）。
- H2：不同故障类型（指标异常/日志报错/链路慢）的 QL 生成难度差异在可接受范围。
- **最小验证方式**：1 人 × 1~2 天。取 20~30 真实故障描述 + 真实 metric 目录，用目标 LLM 跑 few-shot，人工评分准确率与错误类型分布。产出准确率报告 + 典型失败 case 集合。

### #2 历史故障工单与 RCA 语料库（飞轮前提）
**理由**：brainstorm Top1（知识库）与 Top3（剧本库）是产品差异化的根，两者都依赖历史语料；PM-G1 痛点基线也从此提取。一份数据可同时验证"高频模式占比""RCA 可结构化""症状可复用"三个假设，是杠杆最高的调研项。

**待验证假设**：
- H1：Top 10 故障模式覆盖 ≥60% 真实事件（剧本库价值前提）。
- H2：历史 RCA 的"症状->根因"可被结构化且对当下有预测力（知识库价值前提）。
- **最小验证方式**：拉取近 3~6 个月 incident 工单与 RCA，人工分类标注（1~2 天）。产出故障模式分布表 + 可剧本化占比 + 可结构化 RCA 样本。

### #3 量化痛点基线（价值裁判）
**理由**："该不该做"最终由数据回答。无基线则 MVP 完成后无法证明降本增效，也无法设 PM-G4 成功阈值。这是争取资源、说服赞助人、定义 MVP 验收的依据。

**待验证假设**：
- H1：SRE 每月被求助工单中，"深层排查"占比足够高，证明自动化价值空间大。
- H2：当前平均排障耗时显著高于"AI 辅助可达到"的水平，存在可压缩空间。
- **最小验证方式**：从工单系统/On-call 记录提取近 3 个月数据（求助量、分类、处理时长、MTTR）。产出基线数字表。

### #4 真实观测后端环境审计 + 真实数据样本（落地前提）
**理由**：技术设计的 MetricCatalog、QL 校验、prompt 元数据注入、tools 客户端、timeout 策略全部依赖真实 API 与数据。不审计则 ADR-004 的实现是纸上谈兵，单测无法写真实边界用例。

**待验证假设**：
- H1：Mimir/Loki/Tempo API 从 Agent 部署位置可达，鉴权方式与限流可接受。
- H2：真实 metric/label 规模在 LLM context 可承受范围（目录可注入 prompt）。
- **最小验证方式**：1 天。核对 API 版本/端点/鉴权/限流/网络；导出 metric 名清单 + 高频 label + 日志/Trace 样本格式。产出环境审计表 + 数据样本集。

### #5 安全合规与数据敏感性审查（上线门槛）
**理由**：私有数据中心 + LLM 是强合规场景。若 service 名/日志含敏感信息且不能出私网，则必须本地化模型，直接推翻"用 openai SDK 调云端 API"的 ADR-001/006 假设。这是硬门槛，越早确认越能避免返工。

**待验证假设**：
- H1：可观测性数据中不含不可外发的敏感信息，或可经脱敏后外发。
- H2：若必须本地化，目标本地模型（如 vLLM + 开源模型）的 QL 生成能力达标（回 #1 验证）。
- **最小验证方式**：与安全/合规团队 1 次评审 + 抽样数据敏感性评估。产出合规结论 + LLM 部署形态决策。

## 6. 建议的调研执行顺序

```
Week 1（命门验证，可并行）
  ├── #1 LLM QL 可行性 spike   ← 最关键，决定方向
  ├── #5 安全合规审查           ← 决定 LLM 部署形态，反哺 #1
  └── #4 环境审计 + 数据样本    ← 为 #1 提供真实数据输入

Week 2（价值与飞轮验证）
  ├── #2 历史 RCA 语料库        ← 验证知识库/剧本库前提
  └── #3 痛点基线               ← 从同一批工单提取

Week 2 收尾：Discovery Gate
  └── 汇总五项结论，决定 GO / NO-GO / 调整方向，再进入编码
```

**关键原则**：所有调研都用最小成本（人工抽样、低保真原型、单次评审），目的是**证伪而非证实**--用最少代价淘汰或确认最危险的假设，而非追求完整。Week 2 末设一个 Discovery Gate，五项结论汇总后决定是否进入编码阶段。
