# AI SRE Investigator — 产品创意发散 (Brainstorm)

> 阶段：**Initial Discovery（新产品初探）**
> 输入：`docs/PRD.md`（MVP V1.0）
> 方法：PM / Designer / Engineer 三视角各 5 个具体功能创意 → 综合 Top 5 排序 → 逐条附理由与待验证假设

---

## 0. 产品理解与机会确认

| 维度 | 说明 |
|---|---|
| 目标用户 | 一线开发（主）——不懂 PromQL/LogQL/TraceQL；SRE 专家（次）——被"多对一"挤兑、疲于当查日志客服 |
| 核心痛点 | ① 可观测性 QL 认知壁垒高；② 全公司挤兑少数 SRE 的人力灾难；③ 跨系统排查靠肌肉记忆、RCA 沉淀缺失，MTTR 高 |
| 已定 MVP | 自然语言申告 → 自动 QL 转化执行 → 异步推理 Timeline → 结构化 RCA 报告 |
| 明确边界 | 不改线上状态（无自愈/扩缩容/改配置）；不用臃肿框架 |
| 关键风险 | **信任**（AI 误判会被立刻打回 SRE 人工）、**可靠性**（观测后端 5xx/超时导致 Agent 中断）、**冷启动**（新服务无历史画像时 agent 不知从何查起）|

本次发散围绕三个风险 + "如何让产品被组织真正采纳并越用越聪明"展开。

---

## 1. PM 视角：市场契合 / 价值创造 / 差异化

### PM-1 RCA 知识库与模式记忆（Incident Memory）
把每一次成功排查的 RCA（症状 → 查询路径 → 根因）结构化入库，形成组织级故障图谱。下次相似症状命中时，Agent 优先复用历史"黄金排查路径"，甚至直接给出疑似根因。
**为什么**：直击 PRD 痛点 ③"排查缺乏标准沉淀"。让产品**越用越聪明**，这是 SRE 个人经验沉淀为组织资产的关键，也是对"全公司挤兑少数 SRE"最本质的解药——把少数人的肌肉记忆变成全员可调用的资产。

### PM-2 置信度感知的人机协同升级（Confidence-aware Escalation）
Agent 对每次 RCA 输出一个置信度分（基于证据完备度、Trace 是否闭环、是否命中已知模式）。低置信度或证据缺失时，自动生成"交接卡"一键转给 on-call SRE，附上已查到的证据链而非从零开始。
**为什么**：AI 在生产 SRE 场景最大的采纳障碍是"不敢用"。给出明确的"我不确定，交给人"出口，是让组织敢把真实故障交给它的前提，也是 MVP 阶段安全上线的护栏。

### PM-3 黄金路径排查剧本库（Golden-path Playbooks）
为高频故障模式（DB 连接池打满、Pod OOM、下游依赖超时、限流触发、GC 停顿）预置"剧本式"排查流：触发条件 → 标准查询序列 → 判定证据 → 常见根因。Agent 优先套用剧本，未命中再走自由推理。
**为什么**：80/20 法则——少数故障模式覆盖大多数真实事件。剧本让 Agent 在"还不够聪明"的早期就能对高频场景给出可靠答案，是**快速验证价值**的最短路径，也降低了对 LLM 推理质量的单一依赖。

### PM-4 服务画像与归属模型（Service Ownership Graph）
自动采集服务元数据（owner team、依赖拓扑、SLO、历史故障率），让 RCA 自动定位责任方并打标。MVP 阶段先用静态配置 + 服务名映射，后续接入 CMDB/服务目录。
**为什么**：企业级落地必备的治理与权限基座；也让 RCA 能回答"这是谁的问题"，直接缩短事件沟通链路，是 PRD"自己服务自己闭环"的延伸。

### PM-5 投资回报仪表盘（Deflection & MTTR Dashboard）
量化产品自身价值：自闭环率（deflection rate）、平均排查时长、SRE 被打扰次数下降、Token/查询成本。每周自动出价值简报给 SRE 与研发负责人。
**为什么**：Initial Discovery 的核心命题是"这个产品该不该存在"。没有可量化的价值证据，很难争取二期资源；这套指标本身也是 MVP 阶段验证产品假设的仪表盘。

## 2. Designer 视角：体验 / 引导 / 参与度

### D-1 信任优先的证据可下钻 Timeline（Trust-first Evidence Drill-down）
Timeline 不只是"步骤列表"，每一步都是可展开的证据卡：指标图（标注波峰时刻）、日志样本（高亮异常行）、Trace 瀑布图（标红慢 Span）。用户可点击任何证据直接向 Agent 追问"这条 span 为什么慢"。
**为什么**：AI 给出的结论若不可验证，开发宁可回去问 SRE。**透明可追溯是信任的基石**，也是"让开发敢独立闭环"的体验关键。这条把 PRD 的"前端进度流"从展示升级为可交互的取证界面。

### D-2 对话式追问而非重开（Conversational Follow-up）
围绕一个 RCA，用户可继续自然语言下钻："把 payment-service 这段时间的 p99 再拉长 3 小时"、"这个 TraceID 的下游调用展开看看"。Agent 复用已有上下文与缓存结果，而非从头推理。
**为什么**：排查本质是迭代式假设验证，不是一次性问答。MVP 若每次都从零开始，体验割裂、成本翻倍，用户会很快放弃。

### D-3 2 分钟模拟故障上手引导（Simulated First Investigation）
新用户首次进入即用一个**预置的模拟故障**（如 demo 服务的 OOM）跑通完整流程，2 分钟内拿到一份真实 RCA。引导其在真实环境提交第一个问题。
**为什么**：冷启动信任鸿沟 + "不敢用"心理。让用户先在零风险场景体验价值，是破除"AI 排查不靠谱"刻板印象最快的手段。

### D-4 一键分享 RCA（Shareable RCA Permalink）
RCA 报告生成可分享链接（含证据快照，数据脱敏），并支持一键发到 Slack / 钉钉 / PagerDuty 对应 on-call 频道。
**为什么**：直击 PRD 痛点 ③"事后整理 RCA 繁琐耗时"。把"写报告"这一高摩擦动作变成零摩擦，是产品粘性与传播力的放大器。

### D-5 不确定性的人话表达（Honest Uncertainty UX）
当证据不足或推理有歧义时，Agent 在报告里明确标注"⚠️ 此结论证据不足，可能原因 A/B，建议进一步查 X"，而非给一个笃定但可能是错的结论。UI 用置信度色阶（绿/黄/红）呈现。
**为什么**：过度自信的 AI 在 SRE 场景是灾难。诚实的"我不知道"反而建立专业可信感，与 PM-2 的升级机制在体验上闭环。

## 3. Engineer 视角：技术创新 / 集成 / 平台能力

### E-1 QL 预校验 + 失败自修正（Validate-before-Execute + Self-heal Queries）
LLM 生成的 PromQL/LogQL/TraceQL 在执行前先做语法/作用域校验（轻量 parse 或 dry-run）。校验失败时，把结构化错误回灌给 LLM 让其修正，而非直接中断或把 500 抛给用户。
**为什么**：PRD 红线强调"5xx 必须 try-catch 并回灌原因"。把"回灌"从被动容错升级为主动自修正闭环，是 Agent 可靠性的命门——LLM 生成 QL 的错误率天然不低，没有自修正，产品在真实环境会频繁中断。

### E-2 工具结果缓存 + 查询预算控制（Caching & Query Budget）
按 (query, time_range, service) 做结果缓存，避免一次排查内重复打观测后端；单次会话设 token/查询次数/耗时上限，超限优雅收尾而非失控循环。
**为什么**：Mimir/Loki 大范围查询又慢又重，不加缓存与预算，一次排查可能拖垮后端或烧爆成本。这是产品能稳定上生产的前置工程门槛。

### E-3 可插拔工具注册表 + MCP 兼容契约（Pluggable Tool Registry）
当前三个 tool 背后统一抽象为 `ToolSpec(name, schema, handler)` 注册表，参数/返回都用 Pydantic 强类型。设计与 MCP server 的 tool schema 对齐，未来"原生 Tool-Calling → MCP"只需替换 transport 层。
**为什么**：PRD 明确演进方向是 MCP/GOAP。现在不做这层抽象，未来迁移成本巨大；现在做了，几乎零成本拿到可扩展性（新增 Prometheus/ES/k8s 工具只需注册）。

### E-4 流式可恢复会话（Streaming + Resumable Sessions）
用 SSE 流式推送每一步 tool call 思考；会话状态落盘，支持断线重连、长任务后台续跑、用户离开后回来继续看进度。
**为什么**：多步排查天然长耗时，前端挂起/刷新就丢进度会严重打击信任。PRD 已提到 SSE 与轮询，这里补齐"可恢复"，是体验与工程一致性的保障。

### E-5 观测 Agent 自身的可观测性（Self-Observability / Dogfooding）
给 Agent 本身埋点：每次排查的 token 用量、tool 调用次数与延迟、循环深度、成功率、缓存命中率、失败原因分布。用 Mimir/Loki 看自己——**用自家产品排障自家产品**。
**为什么**：受众就是 SRE，dogfooding 是最好的信任背书与质量反馈环；这些指标直接喂给 PM-5 的价值仪表盘，也驱动 E-1/E-2 的持续调优。一举多得，且与产品调性高度契合。

## 5. Top 5 的理由与待验证假设

### #1 RCA 知识库与模式记忆
**理由**：PRD 痛点 ③"排查耗时且缺乏标准沉淀"的本质是——经验只存在于 SRE 个人脑中。RCA 知识库让每一次成功排查都成为组织资产，且 Agent 命中历史模式可大幅缩短排查路径。这是对"多对一挤兑"最根本的解药，也是竞品（纯问答式 AI 运维工具）难以复制的护城河：数据飞轮越转越快。

**待验证假设**：
- H1：真实故障的"症状→根因"模式可被结构化、可复用（即历史对当下有预测力）。若故障高度发散，记忆价值有限。
- H2：开发愿意在拿到 RCA 后花 30 秒做轻量标注/确认根因（数据飞轮的输入端）。若无人标注，库会退化。
- 验证方式：MVP 先用人工整理的 20~30 条历史 RCA 做"检索增强"，看命中率与采纳率，再决定自动化沉淀。

### #2 信任优先的证据可下钻 Timeline
**理由**：SRE 场景里，"AI 说了什么"远不如"AI 凭什么说"重要。可展开、可点击、可追问的证据卡让开发能自行核实结论，是把 SRE 的"举证责任"下放给 AI 的关键。没有这一层，"自助闭环"只是口号；有了它，开发才真正敢签字画押不再求助。

**待验证假设**：
- H1：开发真的会点开证据核验，而非直接采信/直接不信（决定这层交互的投入产出比）。
- H2：证据展开的信息量与"不过载"之间可平衡（太详细劝退、太简略不信任）。
- 验证方式：A/B 对比"纯结论版"与"可下钻版"的自闭环率与回访 SRE 率。

### #3 黄金路径排查剧本库
**理由**：LLM 自由推理的稳定性在早期不足以支撑生产排查。剧本库把"智能"约束在经过 SRE 验证的高频路径上，既保证可靠又快速可验证，还降低了对单一 LLM 能力的依赖。它是 MVP 阶段"用最小聪明换取最大可靠"的杠杆点。

**待验证假设**：
- H1：真实事件中，高频故障模式占比足够高（如 Top 10 剧本覆盖 ≥60% 事件）。若分布长尾，剧本价值有限。
- H2：剧本的"触发条件→标准查询"能被自然语言可靠命中（即意图识别到剧本的准确率）。
- 验证方式：取近 3 个月真实 incident 工单做离线分类，统计可剧本化占比。

### #4 QL 预校验 + 失败自修正 + 缓存/预算
**理由**：这是"能用"与"好用"的分水岭。LLM 生成 QL 错误率不低，观测后端查询又重又慢；没有校验自修正会频繁中断，没有缓存预算会拖垮后端或烧爆成本。PRD 已把容错列为红线，这里把红线从"不崩"升级为"自愈 + 可控"。

**待验证假设**：
- H1：LLM 在拿到结构化错误后，单轮自修正成功率足够高（如 ≥70%），值得做回灌而非直接报错。
- H2：单次排查的查询去重率足够，缓存收益大于其复杂度成本。
- 验证方式：对一批真实问题统计 QL 首次错误率与修正成功率、重复查询占比。

### #5 置信度感知的人机协同升级
**理由**：组织敢不敢把真实故障交给 AI，取决于"AI 在不确定时会不会硬撑"。显式置信度 + 优雅交接卡，是产品从"玩具/辅助"跨入"可信生产工具"的安全闸，也直接降低上线初期的风险暴露，为激进迭代赢得空间。

**待验证假设**：
- H1：置信度能被可信地估算（基于证据完备度等启发式是否与"实际是否正确"相关）。
- H2：低置信度交接卡确实被 SRE 认为"有用、省事"，而非"又多一个要处理的卡"。
- 验证方式：灰度期间记录 SRE 对交接卡的处置率与满意度。

## 6. 给 MVP 的落地建议（非发散，仅供决策参考）

- **必做（MVP 增强项）**：#4 的"QL 预校验 + 自修正 + 容错回灌"是 PRD 红线的工程化落地，应纳入 V1.0；#2 的可下钻证据 Timeline 应作为前端进度流的目标形态。
- **快赢（用最小代价验证假设）**：#3 先人工写 5~10 个剧本做检索增强，验证 H1；#1 先用 20~30 条历史 RCA 验证命中率，再谈自动化沉淀。
- **安全闸**：#5 的置信度 + 交接在灰度期必须有，否则真实事件不敢用。
- **二期**：E-5 dogfooding、PM-5 ROI 仪表盘、E-3 工具注册表抽象。
