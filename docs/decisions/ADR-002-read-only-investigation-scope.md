# ADR-002: 只读排查边界——不修改任何线上状态

## Status
Accepted

## Date
2026-07-09

## Context
AI SRE Investigator 接入生产可观测性平台（Mimir/Loki/Tempo），具备对线上系统的深度洞察力。需要明确 Agent 的动作边界：仅排查（读）还是可处置（写）。

关键考量：
- 生产环境任何写操作（自愈、扩缩容、改配置、重启）都带爆炸半径。
- MVP 阶段首要目标是验证"AI 能否可靠还原故障根因"，而非自动化处置。
- 组织信任是逐步建立的：先证明"读得准"，才有资格谈"写得对"。
- 处置类操作涉及权限、审计、回滚链路，工程与合规复杂度远超读操作。

## Decision
**MVP 严格限定为只读排查，不直接修改集群或线上状态——无自愈、无扩缩容、不改配置。** Agent 只能查询 Mimir/Loki/Tempo，输出 RCA 报告与修复建议（建议给人执行），自身不执行任何变更。

## Alternatives Considered

### 自愈闭环（Auto-remediation）
- Pros：理论上能进一步压缩 MTTR，是 SRE 自动化的终极形态。
- Cons：爆炸半径巨大；AI 误判会导致线上事故扩大；权限、审计、回滚链路复杂；MVP 阶段组织尚未建立对 AI 判断的信任。
- Rejected：信任未建立前贸然开放写权限是灾难。应先以只读证明价值，处置能力作为后续演进。

### 半自动处置（建议 + 一键执行，但需人确认）
- Pros：比纯自愈安全，比纯只读更省事。
- Cons：仍需打通变更执行链路、权限模型、审计日志；MVP 范围膨胀。
- Rejected（暂缓）：可作为 V2 演进方向，但前提是只读 RCA 已建立足够信任与置信度体系（见 brainstorm Top #5 置信度感知升级）。

## Consequences
- 正向：爆炸半径为零，上线风险可控，组织敢在真实事件中试用。
- 正向：无需构建变更执行、权限、审计、回滚链路，MVP 工程量聚焦在排查核心价值。
- 正向：RCA 报告作为"修复建议"产出，仍能显著缩短沟通链路（人执行建议即可）。
- 约束：Agent 不会真正降低"修复"环节耗时，价值上限是"诊断"而非"闭环"——需在产品定位与预期管理上明确。
- 演进口：未来开放处置时，必须先具备 ADR-005 的置信度体系与人工升级闸门，作为写操作的前置安全条件。
